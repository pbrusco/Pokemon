# State Management & FSM

## Overview

Game state is split into two layers:

| Layer | Where | What |
|-------|-------|------|
| **Global persistent state** | `src/store/gameStore.ts` (Zustand) | Position, team, inventory, progression flags |
| **Ephemeral game-phase state** | `src/App.tsx` (React `useState`) | Current mode (exploring, battle, menu), animation states, battle log |

---

## Zustand Store (`src/store/gameStore.ts`)

### State Shape

```typescript
interface GameState {
  // Location & Movement
  playerPos: Position           // { x, y } in tile coordinates
  direction: Direction          // 'up' | 'down' | 'left' | 'right'
  currentMap: string            // e.g. 'PALLET_TOWN'
  isMoving: boolean

  // Story Progression
  storyStep: StoryStep          // See gameplay.md for values
  badges: string[]
  defeatedTrainers: string[]    // NPC IDs of beaten trainers
  hasPokedex: boolean
  hasParcel: boolean
  lastHealLocation: { map: string; pos: Position }

  // Inventories & Teams
  inventory: string[]           // Array of item IDs (e.g. 'POTION')
  playerTeam: Pokemon[]         // Active party (max 6)
  pcStorage: Pokemon[]          // Box storage (unlimited)

  // Mutable World (map editor changes)
  worldMaps: Record<string, Tile[][]>
  teleports: Record<string, Entity[]>
  items: Record<string, Entity[]>

  // Derived
  getNPCs: () => Record<string, NPC[]>   // Computed from progression flags
}
```

### Key Mutations

```typescript
// Movement
store.setPlayerPos(pos: Position)
store.setDirection(dir: Direction)
store.setCurrentMap(map: string)
store.setIsMoving(moving: boolean)

// Progression
store.setStoryStep(step: StoryStep)
store.addBadge(badge: string)
store.addDefeatedTrainer(id: string)
store.setHasPokedex(val: boolean)
store.setHasParcel(val: boolean)
store.setLastHealLocation(loc)

// Team & Inventory
store.updateTeam(team: Pokemon[])
store.updatePcStorage(pc: Pokemon[])
store.addInventoryItem(id: string)
store.removeInventoryItem(id: string)

// Save/Load
store.loadPersistedState(payload)
```

### Initial State

- Player at Pallet Town `(10, 10)`, facing `'down'`
- Inventory: `['POTION', 'POKEBALL']`
- `storyStep: 'START'`

### Dynamic NPC Generation

NPCs are not stored statically — they are generated by calling `getNPCs()`, which calls `generateWorldNPCs(hasParcel, hasPokedex, badges)` from `worldConfig.ts`. This means NPC dialogue and availability automatically reflect story state without manual updates.

## Game Phase FSM (`src/types/gamePhase.ts`)

The game uses a two-level finite state machine stored as a single `useState<GamePhase>` in `App.tsx`. This replaces the old pattern of ~17 independent boolean flags.

### Top-Level `GamePhase`

```typescript
type GamePhase =
  | { type: 'EXPLORING' }
  | { type: 'MENU' }
  | { type: 'INVENTORY' }
  | { type: 'TEAM' }
  | { type: 'SHOP' }
  | { type: 'POKEDEX' }
  | { type: 'PC' }
  | { type: 'EDITOR' }
  | { type: 'BATTLE_TRANSITION' }
  | { type: 'BATTLE'; sub: BattlePhase }
  | { type: 'BLACKOUT' }
  | { type: 'HEALING' }
```

### Battle Sub-Phases (`BattlePhase`)

Only valid when `GamePhase.type === 'BATTLE'`.

```typescript
type BattlePhase =
  | { type: 'CHOOSING' }          // Player picks an action
  | { type: 'PLAYER_ATTACK' }     // Player's move executing
  | { type: 'ENEMY_ATTACK' }      // Enemy's move executing
  | { type: 'PLAYER_FAINTED' }    // Active Pokemon fainted (team still alive)
  | { type: 'FORCED_SWITCH' }     // Player must pick a replacement
  | { type: 'ENEMY_FAINTED' }     // Enemy defeated
  | { type: 'CATCHING' }          // Pokeball thrown, catch animation
  | { type: 'LEVEL_UP' }          // Level-up message displaying
  | { type: 'EVOLVING' }          // Evolution animation
  | { type: 'BATTLE_INVENTORY' }  // Bag opened mid-battle
  | { type: 'BATTLE_TEAM' }       // Team menu opened mid-battle
```

### State Transitions

```
EXPLORING ──────────────────────────────→ MENU / INVENTORY / TEAM / SHOP / POKEDEX / PC / EDITOR
EXPLORING ──────────────────────────────→ BATTLE_TRANSITION
BATTLE_TRANSITION ───────────────────────→ BATTLE (sub: CHOOSING)

BATTLE CHOOSING ─────────────────────────→ PLAYER_ATTACK
BATTLE PLAYER_ATTACK ────────────────────→ ENEMY_ATTACK
BATTLE ENEMY_ATTACK ─────────────────────→ CHOOSING  (normal loop)
BATTLE ENEMY_ATTACK ─────────────────────→ PLAYER_FAINTED → FORCED_SWITCH
BATTLE PLAYER_ATTACK ────────────────────→ ENEMY_FAINTED → LEVEL_UP → EVOLVING
BATTLE CHOOSING ─────────────────────────→ BATTLE_INVENTORY / BATTLE_TEAM

BATTLE (any) ────────────────────────────→ EXPLORING  (battle ends)
BATTLE (loss) ───────────────────────────→ BLACKOUT → EXPLORING
```

### Helper Constants & Factory

```typescript
// Pre-built instances for common phases
export const EXPLORING: GamePhase = { type: 'EXPLORING' };
export const MENU: GamePhase = { type: 'MENU' };
export const BATTLE_TRANSITION: GamePhase = { type: 'BATTLE_TRANSITION' };

export const B_CHOOSING: BattlePhase = { type: 'CHOOSING' };
export const B_PLAYER_ATTACK: BattlePhase = { type: 'PLAYER_ATTACK' };
export const B_ENEMY_ATTACK: BattlePhase = { type: 'ENEMY_ATTACK' };
// ...etc

// Factory: creates a BATTLE phase with a given sub-phase
export function battle(sub: BattlePhase): GamePhase {
  return { type: 'BATTLE', sub };
}
```

### Usage in App.tsx

```typescript
const [phase, setPhase] = useState<GamePhase>(EXPLORING);

// Derived helpers
const inBattle = phase.type === 'BATTLE';
const battlePhase = phase.type === 'BATTLE' ? phase.sub : null;

// Transitions
setPhase(BATTLE_TRANSITION);           // start the battle animation
setPhase(battle(B_CHOOSING));          // enter the choosing state
setPhase(battle(B_FORCED_SWITCH));     // force a team swap after faint
setPhase(EXPLORING);                   // return to overworld
```

### Why This Pattern?

Before the FSM, the game had ~17 booleans: `isBattle`, `isChoosingMove`, `isPlayerAttacking`, `isEnemyAttacking`, `isLevelingUp`, `isEvolving`, `isCatching`, `showInventory`, `showTeam`, etc.

Problems with booleans:
- Multiple could be `true` simultaneously (impossible states)
- Every transition required setting several flags
- Conditional rendering was a chain of `&&` checks

With the FSM, impossible combinations are rejected by TypeScript at compile time.
